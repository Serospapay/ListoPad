# Аудит поточного стану проєкту `ЛистоПад`

Дата аудиту: 2026-04-15  
Формат: фактичний технічний зріз реалізованого стану без припущень.

## 1) Що це за проєкт

Монорепозиторій з двома частинами:

- Frontend: SPA на `React 19 + TypeScript + Vite`.
- Backend: `Django + Django REST Framework` (окремий API-сервер).

Загальна предметна область: інтернет-магазин/видавництво книг з публічним каталогом, особистим кабінетом користувача та базовими admin-екранами (замовлення, CRM, склад).

## 2) Технічна реалізація (архітектурно)

### Frontend

- Єдина точка маршрутизації в `App.tsx` через локальний `activeView` (без React Router).
- Дані (книги, користувачі, замовлення, категорії) завантажуються через `services/api.ts`.
- При недоступному API вмикається fallback на `localStorage` (ключі `lystopad_*`).
- Стейт у `App.tsx`: авторизація, кошик, wishlist, список книг, список користувачів, замовлення, категорії, тема.
- UI побудований переважно Tailwind-класами, підключення Tailwind через CDN в `index.html`.
- Іконки: Font Awesome CDN.

### Backend

- API реалізовано через DRF `ModelViewSet`.
- Моделі: `Book`, `Category`, `Order` + використання стандартного `django.contrib.auth.models.User`.
- Ендпоінти через DRF router:
  - `GET/POST/PUT/DELETE /api/books/`
  - `GET/POST/PUT/DELETE /api/categories/`
  - `GET/POST/PUT/DELETE /api/orders/`
  - `GET /api/users/` (тільки read-only)
- Книга підтримує many-to-many категорії (`categories`), у `BookViewSet` є автостворення категорій при create/update.

## 3) Практично реалізований функціонал

### Публічна вітрина

- Перегляд каталогу книг плиткою.
- Пошук за назвою/автором.
- Фільтр за жанром.
- Фільтр діапазону ціни.
- Фільтр "тільки в наявності".
- Сортування за ціною та кількістю сторінок.
- Пагінація каталогу.
- Додавання в wishlist.
- Додавання в кошик.

### Сторінка книги

- Деталі книги (опис, характеристики, ціна, наявність).
- Галерея з перемиканням слайдів.
- Lightbox перегляд зображення.
- Показ схожих книг (за перетином жанрів).
- Перехід за жанром назад у вітрину.

### Авторизація/акаунт

- Екран входу/реєстрації.
- Вхід через API (`apiService.login`) з fallback-логікою на локальні дані.
- Збереження поточного користувача в `localStorage`.
- Профіль користувача:
  - базова картка профілю;
  - wishlist;
  - таблиця історії замовлень (із статусами);
  - вихід з акаунта.

### Checkout

- Кошик із можливістю змінювати кількість і видаляти позиції.
- Вибір способу доставки і способу оплати.
- Підрахунок суми, доставки і total.
- Екран "успішно" після оформлення.

### Адмінські екрани

- `Склад`:
  - перегляд книг;
  - пошук;
  - фільтр дефіциту (inventory < 10);
  - додавання/редагування/видалення книги;
  - керування жанрами (додавання/перейменування/видалення);
  - завантаження до 10 фото, компресія на клієнті.
- `CRM`:
  - агрегована статистика (дохід, кількість користувачів, активні замовлення);
  - таблиця користувачів з пошуком;
  - модалка з історією замовлень клієнта;
  - графік продажів (recharts).
- `Реєстр замовлень`:
  - групування замовлень за клієнтом;
  - перегляд деталей;
  - зміна статусу замовлення через select.

### Інформаційні сторінки

- `Контакти`, `Доставка`, `Оплата` присутні як окремі екрани.
- Є технічна заготовка контенту і структури блоків.

## 5) Що працює частково або неповно (фактичні обмеження)

### Критичні функціональні розриви

- Немає завершеного E2E ланцюга оформлення:
  - checkout не створює `Order` на backend;
  - підтвердження платежу і замовлення імітується через `setTimeout`.
- В `apiService.login` використовується `/api/users/login/`, але такого endpoint у DRF router немає.
- Реєстрація користувача виконується тільки локально в frontend (без API і без запису в БД Django).
- Зміна статусу замовлення в `AdminOrders` міняє локальний state, але не зберігає в backend.

### Узгодженість даних

- Для частини дій дані зберігаються тільки в `localStorage` (`saveBooks`, `saveCategories`), без синхронізації через API.
- У `apiService` майже всі помилки в `catch` ковтаються без логування/повернення причини.

### Контент і UX

- На сторінках `Contacts`, `Delivery`, `Payment`, а також у частині checkout/book details є плейсхолдери `ячс`/технічний текст.
- Є дрібні неузгоджені написи ("дородчих", "Видавництво???", "Доступно: ??").

## 7) Фактичний рівень готовності по підсистемах

- UI/UX-концепт: **високий** (дизайн і компоненти покривають основні екрани).
- Каталог/вітрина: **високий**.
- Склад (books/categories): **середній-високий**.
- CRM/аналітика: **середній** (працює на поточних даних, частково демонстраційний характер графіка).
- Auth: **середній-низький** (є UI і часткова логіка, але немає повного backend auth-flow).
- Checkout/Order E2E: **низький-середній** (UI є, транзакційна частина неповна).
- Продакшн-безпека/експлуатація: **низький**.

## 8) Нереалізовані/непідключені артефакти в коді

- `components/AdminDashboard.tsx` присутній, але не використовується в `App.tsx`.
- `services/analytics.ts` присутній, але не використовується в основному потоці.
- У залежностях є `@supabase/supabase-js` та `@google/genai`, але в робочому коді інтеграції фактично не задіяні.

## 9) Рекомендований технічний пріоритет (коротко)

1. Закрити E2E checkout: `POST /orders`, списання inventory, оновлення історії замовлень у профілі без перезавантаження.
2. Додати справжню auth-схему (login/register/logout) з backend endpoint-ами і валідацією.
3. Прибрати fallback-критичність localStorage для серверних сутностей, зробити чітку політику sync/offline.
4. Добити контент сторінок `Contacts/Delivery/Payment`, прибрати плейсхолдери.
5. Вивести backend у production-safe конфіг (env-based settings, CORS, secret, debug off).
